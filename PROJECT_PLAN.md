# OpenBible 项目执行计划

更新日期：2026-07-23

## 1. 项目目标

构建以圣经阅读、注释、慧读与团体社群为核心的移动端产品，并提供管理员后台、成员权限、内容审核、活动、小组、资料库及后续付费会员能力。

第一阶段以“可用、稳定、低成本”为优先，不在未确认预算前开通任何付费云服务或正式发布渠道。

## 2. 产品结构

- 圣经：多译本阅读、章节选择、语音、字号、滑动换章、经文选择、笔记、高亮、分享。
- 注释：卷章选择、绪论、段落综览、逐节注释、从经文精确定位。
- 慧读：个人对话、历史记录、经文上下文、内容安全提示。
- 社群：群动态、助手、活动、更多四个一级栏目；群动态作为默认内容入口，成员、小组、资料归入“更多”。
- 我的：个人/团体账户、登录、笔记、收藏、设置、会员状态。
- 管理后台：群组、成员与角色、内容审核、活动、资料、会员及运营数据。

## 3. 社群栏目职责

一级导航保持聚焦：

| 一级栏目 | 用户能力 | 管理能力 |
| --- | --- | --- |
| 群动态 | 发布图文、文章与影音，点赞和评论 | 发布通知，审核、隐藏和删除 |
| 助手 | 私密问答、生活关怀、快捷查找社群内容 | 经用户确认后协助办理社群事务 |
| 活动 | 浏览、报名、取消报名和查看日程 | 创建活动、名额和报名管理 |
| 更多 | 进入成员、小组和资料 | 右上角设置进入社群管理 |

“更多”页面承载三个二级功能：

| 二级功能 | 用户能力 | 管理能力 |
| --- | --- | --- |
| 成员 | 查看成员与角色 | 审批、设置管理员和移除成员 |
| 小组 | 浏览并进入子小组 | 创建和管理下属小组 |
| 资料 | 浏览文档、链接、音频和视频 | 设置可见范围和下架资料 |

当前版本已经完成真实数据库闭环：群动态支持图文、长文章、管理员通知、图片/音频/视频链接、分类筛选、点赞、评论、收藏和举报；通知支持置顶；群主与管理员可隐藏或删除内容，所有管理操作由后端校验权限并写入审计日志。活动支持创建/报名；成员支持角色与加入审批；小组使用父子社群；资料支持分类链接。当前媒体采用外部 `http/https` 链接，不产生新增文件存储费用。后续增强按优先级继续：

1. 群动态：真实文件上传、分页、举报处理状态和“我的收藏”聚合页。
2. 活动：编辑/取消、提醒、签到、报名名单和活动回顾。
3. 成员：搜索、邀请、公开资料、活跃情况、禁言与群主转让。
4. 小组：申请加入/退出、归档、负责人和成员调组。
5. 资料：分类、搜索、收藏、下载记录和真实文件上传。

## 4. 权限基线

- 访客：浏览公开内容。
- 普通成员：发布、互动、报名、申请加入小组。
- 小组管理员：管理所属小组成员与内容。
- 群管理员：管理整个群的成员、内容、活动、小组与资料。
- 群主：拥有管理员任免、付费方案与高风险操作权限。
- 平台管理员：处理平台级审核、安全、运营和系统配置。

高风险操作必须二次确认，并在后端再次校验权限和写入审计日志。

## 5. 执行阶段与验收

### 阶段 0：基线与成本控制（1–2 个开发日）

- 固化功能清单、权限矩阵、接口契约、数据模型和验收用例。
- 建立环境变量清单、测试账号、错误反馈和成本计量点。
- 验收：范围无冲突、角色权限可测试、付费资源默认关闭。

### 阶段 1：社群核心闭环（3–5 个开发日）

- 分享、活动、成员接入真实 API 和数据库。
- 完成加载、空态、失败、重试、分页和权限反馈。
- 验收：成员可发布与互动，管理员可审核，活动可完整报名。

### 阶段 2：小组与资料库（4–6 个开发日）

- 使用父子社群关系实现小组。
- 增加资料、分类、可见范围、收藏和下载记录的数据模型。
- 验收：权限隔离正确，小组和资料均可由后台维护。

### 阶段 3：团体管理与会员基础（3–5 个开发日）

- 完善个人/团体账户、角色管理、内容审核和审计日志。
- 建立免费版/团体会员的权益与配额模型，不直接接入收费。
- 验收：权益由后端判定，管理员页面和普通成员页面清晰区分。

### 阶段 4：全站体验与质量（3–5 个开发日）

- 统一头部、图标、按钮、字号、间距、卡片、弹层和反馈。
- 完成核心单元测试、接口测试、移动端流程测试、安全和性能检查。
- 验收：核心流程无阻断问题，主要页面在目标手机尺寸正常显示。

### 阶段 5：内测与上线准备（2–4 个开发日）

- 小规模真实用户内测、数据备份、监控、隐私和发布材料。
- 根据真实使用量复核服务器与 AI 成本，再决定是否扩容和上架。

预计 MVP 工程工作量：16–27 个开发日。实际进度取决于数据迁移、现有接口质量和验收反馈。

## 6. 预算原则

- 本地开发：不增加外部基础设施费用。
- 内测：优先复用现有服务器、免费事务邮件和按量 AI API。
- AI：设置每群、每用户和每日额度，记录 token 与失败重试，达到预算阈值后停止或降级。
- 文件：先限制类型、尺寸和总容量，再按实际容量启用对象存储。
- 正式收费、应用商店上架或新增云资源前，单独提交费用表并取得确认。

## 7. 项目推进规则

- 每个阶段先给出本阶段范围与预计新增费用，再实施和验证。
- 每次交付附带完成项、测试结果、已知问题和下一阶段计划。
- 不以界面“看起来完成”作为验收；必须有数据库、权限、错误处理和测试闭环。
- 不覆盖未确认的本地修改，不提交密钥，不在前端依赖权限隐藏。
